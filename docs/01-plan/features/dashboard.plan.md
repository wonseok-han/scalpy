# Dashboard Plan

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 서버 로그만으로는 트레이딩 상태 파악이 어렵고, 긴급 상황 시 대응 수단이 없음 |
| **Solution** | FastAPI 기반 Web 대시보드 + 텔레그램 알림 봇 |
| **Function UX** | 브라우저에서 포지션/손익/스크리닝/전략 신호를 실시간 확인하고, 긴급 정지/전체 청산 가능 |
| **Core Value** | 자동매매 시스템의 가시성과 제어력 확보 — 방치 운영 시에도 텔레그램으로 핵심 이벤트 수신 |

## Context Anchor

| Key | Value |
|-----|-------|
| **WHY** | 로그 기반 모니터링은 실시간성 부족, 긴급 제어 불가 |
| **WHO** | 개인 투자자 (프로젝트 소유자) — 외출 중에도 모니터링 필요 |
| **RISK** | 대시보드 API가 트레이딩 엔진 성능에 영향을 줄 수 있음 |
| **SUCCESS** | 브라우저에서 실시간 포지션 확인 + 긴급 정지 가능 + 텔레그램 알림 수신 |
| **SCOPE** | FastAPI 서버 + 프론트엔드 + 텔레그램 봇 + 엔진 이벤트 연동 |

## 1. Requirements

### 1.1 대시보드 화면

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| R-1 | 포지션 현황: 보유 종목, 수량, 평균단가, 현재가, 미실현 손익(금액/%) | Must |
| R-2 | 스크리닝 현황: 선별 종목 목록, 등락률, 거래량, 다음 스크리닝까지 남은 시간 | Must |
| R-3 | 전략 신호 로그: 매수/매도 신호 발생 이력, 전략명, 종목, 가격, 시간 | Must |
| R-4 | 계좌 요약: 잔고, 총 평가금액, 당일 실현 손익 | Must |
| R-5 | 엔진 상태: 실행 중/정지, 연결 상태, 마지막 틱 시간 | Should |

### 1.2 제어 기능

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| R-6 | 전체 청산: 모든 보유 포지션 시장가 일괄 매도 | Must |
| R-7 | 엔진 정지/재시작: 자동매매 엔진 일시 정지 및 재개 | Must |
| R-8 | 스크리닝 수동 실행: 즉시 재스크리닝 트리거 | Should |

### 1.3 텔레그램 알림

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| R-9 | 주문 체결 알림: 매수/매도 체결 시 종목, 가격, 수량 전송 | Must |
| R-10 | 손절/익절 알림: 자동 청산 발생 시 즉시 알림 | Must |
| R-11 | 엔진 상태 알림: 시작/종료/에러 발생 시 알림 | Should |
| R-12 | 일일 리포트: 장 마감 후 당일 거래 요약 | Could |

### 1.4 비기능 요구사항

| ID | 요구사항 |
|----|----------|
| NR-1 | 트레이딩 엔진 성능에 영향 없음 (별도 스레드/비동기) |
| NR-2 | 기존 엔진 코드 최소 수정 (이벤트 방식으로 연동) |
| NR-3 | 단일 사용자 (인증 불필요, 로컬 네트워크 전제) |
| NR-4 | 프론트엔드는 가볍게 (SPA 프레임워크 없이 가능하면 vanilla/htmx) |

## 2. Architecture Overview

```
TradingEngine ──(events)──> EventBus ──> DashboardAPI (FastAPI)
                                    ──> TelegramNotifier
                                    
Browser ──(HTTP/SSE)──> DashboardAPI ──(read)──> EngineState
                                     ──(action)──> Engine control
```

- **EventBus**: 엔진 내부 이벤트(체결, 신호, 포지션 변경)를 구독자에게 전파
- **DashboardAPI**: FastAPI 서버, SSE로 실시간 데이터 푸시
- **TelegramNotifier**: EventBus 구독자로 알림 전송

## 3. Success Criteria

| ID | 기준 | 측정 방법 |
|----|------|-----------|
| SC-1 | 브라우저에서 포지션/손익 실시간 확인 가능 | SSE로 1초 이내 반영 |
| SC-2 | 전체 청산 버튼 동작 | 대시보드에서 클릭 → 전 종목 매도 주문 확인 |
| SC-3 | 엔진 정지/재시작 동작 | 대시보드에서 토글 → 엔진 상태 변경 확인 |
| SC-4 | 텔레그램 체결 알림 수신 | 주문 체결 시 5초 이내 텔레그램 메시지 도착 |
| SC-5 | 엔진 성능 무영향 | 대시보드 유무에 따른 틱 처리 지연 차이 < 1ms |

## 4. Risk

| 리스크 | 확률 | 영향 | 대응 |
|--------|------|------|------|
| FastAPI 서버가 엔진 이벤트루프 블로킹 | 중 | 높 | 별도 스레드에서 uvicorn 실행 |
| 텔레그램 API 호출 지연이 엔진에 전파 | 중 | 높 | 비동기 fire-and-forget 방식 전송 |
| SSE 연결 끊김 시 데이터 유실 | 낮 | 중 | 클라이언트 자동 재연결 + 전체 상태 재전송 |
| 전체 청산 중 일부 주문 실패 | 낮 | 높 | 실패 종목 재시도 + 결과 표시 |

## 5. Scope

### In Scope
- FastAPI 대시보드 서버 (포지션, 스크리닝, 신호 로그, 계좌 조회)
- 제어 API (전체 청산, 엔진 정지/재시작, 수동 스크리닝)
- SSE 기반 실시간 업데이트
- 텔레그램 봇 알림 (체결, 손절/익절, 엔진 상태)
- EventBus 모듈 (엔진↔대시보드/텔레그램 연결)
- 경량 프론트엔드 (HTML + htmx or vanilla JS)

### Out of Scope
- 사용자 인증/권한 관리
- 수동 매수/매도 주문 (전체 청산만 지원)
- 차트/캔들스틱 시각화
- 모바일 앱
- 거래 이력 DB 저장 (메모리 내 최근 N건만 유지)
