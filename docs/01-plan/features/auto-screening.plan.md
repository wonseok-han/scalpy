# Auto Screening Plan

> 스캘핑 적합 종목을 자동 선별하여 수동 종목 설정 없이 완전 자동 매매를 실현하는 기능.

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 현재 `settings.toml`에 종목코드를 수동 입력해야 하며, 스캘핑에 적합한 종목을 사용자가 직접 판단해야 함 |
| **Solution** | 거래량·변동성 기반 자동 스크리닝 모듈을 추가하여 장 시작 시 + 주기적으로 종목을 자동 선별 |
| **Function UX Effect** | 서버 구동만 하면 종목 선정부터 매매까지 완전 자동화 달성 |
| **Core Value** | 스캘핑 기회를 놓치지 않는 실시간 종목 탐색 + 사람의 판단 편향 제거 |

## Context Anchor

| Key | Value |
|-----|-------|
| **WHY** | 수동 종목 설정은 스캘핑 기회를 놓치게 하고, 완전 자동화를 가로막는 병목 |
| **WHO** | 개인 투자자 (프로젝트 소유자) — 서버 구동 후 방치형 자동매매 희망 |
| **RISK** | KIS API 호출 제한 (초당 20건), WebSocket 구독 20종목 제한, 스크리닝 품질이 수익에 직결 |
| **SUCCESS** | 수동 종목 설정 없이 자동 종목 선별 → 전략 신호 → 매매 파이프라인 완성 |
| **SCOPE** | Screener 모듈 신규 + TradingEngine/main.py 연동. 전략/브로커 모듈은 변경 없음 |

## 1. 배경 및 목적

현재 Scalpy는 `config/settings.toml`의 `symbols` 리스트에 종목코드를 수동 입력해야 한다.
스캘핑은 **거래량이 폭증하고 변동성이 높은 종목**에서 효과적이므로, 이를 자동으로 탐색하는 기능이 필수적이다.

목표: **서버 구동 → 종목 자동 선별 → WebSocket 구독 → 전략 실행 → 매매** 의 완전 자동화 파이프라인 완성.

## 2. 요구사항

### 2.1 기능 요구사항

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| FR-1 | 거래량 상위 종목 조회 (KIS REST API) | Must |
| FR-2 | 변동성(등락률) 기준 필터링 | Must |
| FR-3 | 스크리닝 결과로 WebSocket 구독 종목 동적 교체 | Must |
| FR-4 | 장 시작 시 초기 스크리닝 실행 | Must |
| FR-5 | 30분~1시간 주기 재스크리닝 | Should |
| FR-6 | 동시 트레이딩 종목 수 5개 제한 | Must |
| FR-7 | 포지션 보유 중인 종목은 스크리닝에서 제외하지 않음 (청산까지 유지) | Must |
| FR-8 | 스크리닝 기준값 설정 파일에서 조정 가능 | Should |

### 2.2 비기능 요구사항

| ID | 요구사항 |
|----|----------|
| NFR-1 | KIS API 초당 호출 제한 준수 (초당 20건) |
| NFR-2 | 스크리닝 실패 시 기존 종목 유지 (fallback) |
| NFR-3 | 모의투자 환경에서 동일하게 동작 |

## 3. 스크리닝 기준

### 3.1 1차 필터: 거래량

- KIS REST API로 **거래량 상위 종목** 조회 (당일 기준)
- 상위 30~50종목을 후보군으로 추출

### 3.2 2차 필터: 변동성

- 당일 등락률 (고가-저가)/시가 기준
- 변동성이 일정 임계값 이상인 종목만 통과
- 기본 임계값: 2% (설정 가능)

### 3.3 3차 필터: 스캘핑 적합성

- 호가 스프레드가 너무 넓지 않은 종목 (유동성 확보)
- 시가총액 하한 설정 (극소형주 제외)
- 최종 5종목 선별 (거래량×변동성 복합 점수 상위)

## 4. 동작 흐름

```
[장 시작] → Screener.scan()
  ↓
거래량 상위 조회 (KIS REST)
  ↓
변동성 필터링
  ↓
상위 5종목 선별
  ↓
MarketDataStream.update_subscriptions(new_symbols)
  ↓
TradingEngine에 종목 반영
  ↓
[30분~1시간 후] → Screener.scan() 반복
  ↓
포지션 보유 종목은 유지 + 나머지 교체
```

## 6. 성공 기준

| ID | 기준 | 검증 방법 |
|----|------|-----------|
| SC-1 | `settings.toml`에 종목코드 없이 자동 선별된 종목으로 매매 동작 | 모의투자 환경 구동 테스트 |
| SC-2 | 스크리닝 결과 5종목 이내 선별 | 로그 확인 |
| SC-3 | 주기적 재스크리닝 시 WebSocket 구독 동적 교체 | 로그에서 구독/해지 확인 |
| SC-4 | 포지션 보유 종목은 교체되지 않음 | 포지션 있는 상태에서 재스크리닝 테스트 |
| SC-5 | API 호출 제한 초과하지 않음 | 에러 로그 없음 확인 |

## 7. 리스크

| 리스크 | 영향 | 대응 |
|--------|------|------|
| KIS 거래량 상위 API가 모의투자에서 미지원 | 스크리닝 불가 | 실투자 API로 조회 후 모의투자로 매매 (하이브리드) |
| 스크리닝 품질 저하 (부적합 종목 선별) | 손실 | 손절 로직 + 보수적 임계값 기본 설정 |
| 종목 교체 시 기존 WebSocket 끊김 | 시세 유실 | 기존 구독 해지 → 신규 구독 순차 실행 |
| API 호출 제한 초과 | 일시 차단 | sleep 간격 확보 + 재시도 로직 |
